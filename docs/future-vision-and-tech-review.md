## 1. Dify / RAGFlow の評価と方針

### 現状

AI サーバーおよび自宅 PC の WSL2 上で Dify と RAGFlow が稼働中。

### 評価

| 項目 | Dify | RAGFlow |
|------|------|---------|
| 用途 | ノーコード AI ワークフロー構築 | RAG ベースの文書検索・QA |
| 強み | 柔軟なフロー設計、GUI操作 | 文書のチャンク分割・ベクトル検索 |
| 社内展開の課題 | プロンプト設計・フロー構築に専門知識が必要 | ベクトルDB・チャンク設計の理解が必要 |

### 方針

- **一般ユーザーへの直接開放は見送り**: 金属加工の中小企業の現場スタッフに「自分でAIフローを構築してもらう」のは能力・時間の面で現実的でない
- **開発者のプロトタイピング用途として残す**: 「この業務にAIを使えるか？」を素早く試す実験環境として価値あり
- **RAGFlow の技術は将来活用の余地あり**: 社内マニュアル・技術文書の検索機能をカスタムアプリに組み込む場合に、RAG の知見が活きる
- **リソース競合時は優先的に停止**: GPU メモリや CPU リソースが逼迫した場合、Dify / RAGFlow を停止してカスタムアプリを優先

### 結論

> 汎用 AI プラットフォームを全員に使わせるより、業務に特化したアプリを開発者が作って配るほうが、中小企業では圧倒的に機能する。

## 2. Unity 導入の検討結果

### 検討内容

- PPTX Generator プロジェクトへの Unity 導入による UI 改善
- 社内業務ダッシュボードへの Unity 採用

### 結論: 現時点では不採用

#### このプロジェクトに不向きな理由

- 「フォーム入力 → AI処理 → ファイル出力」は典型的な業務 Web アプリであり、HTML/CSS/JS が最適
- WebGL ビルドは初回ロードが数MB～数十MB、社内利用で毎回の待ち時間が発生
- ファイル操作（ドラッグ&ドロップ、ダウンロード）が Unity WebGL では煩雑
- テキストレンダリングが HTML に比べ貧弱（フォント、コピペ、選択）
- 開発・保守コストが跳ね上がる（C# + Unity Editor）

#### 業務ダッシュボードにも不向きな理由

- DB 接続や REST API 呼び出しが Web 技術なら数行で済むが Unity では煩雑
- テーブル表示、テキスト検索、フィルタリング、印刷、CSV 出力が苦手
- 「F1 カーで通勤する」ようなオーバースペック

#### Unity が活きるユースケース（将来参考）

- 倉庫の 3D デジタルツイン
- VR/AR での現場利用（HoloLens 等）
- 大規模リアルタイム可視化（数万ポイントのパーティクル）
- 経営陣向け大画面ショーケース

#### UI 改善の正しいアプローチ

| 選択肢 | 工数 | 効果 |
|--------|------|------|
| 現行の HTML/CSS/JS を磨く | 小 | 高 |
| React / Vue.js | 中 | 高 |
| Tailwind CSS + shadcn/ui | 小～中 | 高 |
| Three.js / D3.js（Web3D） | 中 | 中～高 |
| Unity | 大 | 低（この用途） |

## 3. リアルタイムダッシュボード構想

### 想定する表示内容

- 出荷情報のリアルタイムモニター
- 売上データ解析（日次・月次）
- 在庫統計の傾向グラフ
- KPI カード（前日比、達成率など）

### 推奨技術スタック

```
[基幹DB / Oracle 12] → [FastAPI (WebSocket + REST)] → [React ダッシュボード]
```

#### フロントエンド

| カテゴリ | ライブラリ | 用途 |
|----------|-----------|------|
| UI フレームワーク | React | コンポーネント単位の部分更新 |
| チャート | Recharts / Nivo / D3.js | グラフ・チャート描画 |
| ダッシュボード UI | Tremor / shadcn/ui | KPI カード、トレンドライン |
| 地図 | React-Leaflet + Deck.gl | 出荷ルートのマップ表示 |
| アニメーション | Framer Motion | 数値変動時のスムーズ演出 |
| スタイリング | Tailwind CSS | 高速でモダンなデザイン |

#### イメージ

- 画面上部: KPI カードがリアルタイムでカウントアップ
- 中央: 時系列グラフが自動更新でスルスルと伸びる
- 右側: 出荷先ランキングがアニメーション付きで順位変動
- 下部: 直近の出荷ログがフィード形式で流れる

### 既存 AI サーバーとの共存

- ポート分離: PPTX アプリ `:8000`、ダッシュボード `:3000`
- Docker Compose でサービス追加
- 将来は Nginx リバースプロキシでパス振り分け
- GPU はダッシュボードでは使わないためリソース競合なし

## 4. BOM システムとの連携

### 現状

社内で BOM システムを別案件として開発検討中。Oracle 12 上の基幹データと連携予定。

### BOM データ構成（想定）

```
[BOM データベース]
  ├── 製品マスタ（完成品）
  ├── 部品マスタ（材料・購入品・加工部品）
  ├── 構成表（親子関係・数量・工程）
  ├── 加工条件（工程・設備・時間）
  └── 仕入先・単価情報
```

### 今回のプロジェクトとの連携シナリオ

#### Phase 1: すぐにできる

| 連携 | 内容 |
|------|------|
| BOM → データ分析 | BOM データを AI が部品点数の傾向、コスト構成比、共通部品率を分析 |
| BOM → Excel 出力 | 製品別の原価積算表、部品一覧表を整形して出力 |
| BOM → PPTX 出力 | 新製品の構成説明資料、原価報告書を自動生成 |

#### Phase 2: 中期的な連携

| 連携 | 内容 |
|------|------|
| BOM + 受注データ → 所要量計算 | MRP 的な展開を AI が補助 |
| BOM + 過去実績 → 見積支援 | 類似製品の BOM から加工時間・原価を AI が推定 |
| BOM + 在庫データ → 欠品予測 | 受注受付時に不足部材を即座に表示 |

#### Phase 3: ダッシュボード統合

| 連携 | 内容 |
|------|------|
| BOM → リアルタイム在庫 | React ダッシュボードで在庫・所要量を可視化 |
| BOM → 納期回答 | BOM + 在庫 + リードタイム → 出荷可能日の自動算出 |

### BOM システムの技術選定方針

| 要素 | 推奨 | 理由 |
|------|------|------|
| バックエンド | FastAPI | 今回と同じスキルが活きる |
| データベース | PostgreSQL or Oracle 12（既存） | BOM の親子構造やリレーションに強い |
| フロントエンド | React | BOM 構成のツリー表示、インタラクティブ編集 |
| 認証 | 簡易ユーザー管理 | 社内利用に限定 |

## 5. Oracle 12 基幹システム連携

### 接続方式

| ライブラリ | 特徴 |
|-----------|------|
| **python-oracledb**（推奨） | Oracle 公式。cx_Oracle の後継。Thin モード対応 |
| cx_Oracle | 従来の定番。Oracle Instant Client が必須 |

### AI エージェント用ユーザー（準備中）

- **SELECT 専用**の読み取り専用ユーザーを Oracle 側に作成予定
- INSERT / UPDATE / DELETE 権限は一切付与しない → 基幹データを壊すリスクをゼロに

#### セキュリティのベストプラクティス

```
[生テーブル] → [分析用 VIEW] → [AI_AGENT ユーザー]
                 ↑ ここで整形・カラム制限
```

- 生テーブルを直接見せず、分析用の VIEW を間に挟む
- 不要カラム（個人情報等）を VIEW から除外
- テーブル構造変更時も VIEW で吸収 → AI アプリ側の修正が減る
- 接続元を AI サーバーの IP に制限（リスナー設定 / ファイアウォール）

### FastAPI 側の設定（将来追加予定）

```env
ORACLE_DSN=192.168.1.xxx:1521/ORCL
ORACLE_USER=AI_AGENT
ORACLE_PASSWORD=****
```

### Oracle 12 の注意事項

- Oracle 12 は 2022 年にプレミアサポート終了済み
- セキュリティパッチが出ないため、ネットワーク的に外部から隔離されていることが重要
- 将来の 19c / 23ai へのアップグレードも視野に

### Oracle 連携で実現できること

| 機能 | 内容 |
|------|------|
| CSV エクスポート不要化 | DB から直接取得ボタンで AI 分析 |
| 売上分析 | 「今月の売上を分析して」→ Oracle から即取得 → AI 分析 |
| 在庫レポート | リアルタイムデータで報告書自動生成 |
| 受注傾向予測 | 過去データから AI が季節性・顧客傾向を分析 |

## 6. WSL2 環境の現状と活用方針

### 自宅 PC

| 項目 | 内容 |
|------|------|
| OS | Windows + WSL2 (Ubuntu) |
| Docker | Docker Engine + Docker Compose |
| 稼働中サービス | Dify, RAGFlow, Ollama |
| GPU | なし（CPU モード） |

### 会社 AI サーバー（192.168.1.10）

| 項目 | 内容 |
|------|------|
| OS | Windows + WSL2 (Ubuntu) |
| メモリ | 64GB |
| GPU | RTX 5080 16GB |
| Docker | Docker Engine + Docker Compose |
| 稼働予定 | PPTX Generator, Ollama (Qwen 3.5), ダッシュボード（将来） |

### リソース配分方針

| 優先度 | サービス | リソース |
|--------|---------|---------|
| 高 | カスタム AI アプリ（PPTX Generator 等） | FastAPI は CPU/メモリ軽量 |
| 高 | Ollama (Qwen 3.5 / Llama) | GPU 16GB の大部分を使用。アイドル時は解放される |
| 中 | BOM システム（将来） | CPU/メモリのみ |
| 低 | Dify / RAGFlow | 実験用。競合時は停止 |

### WSL2 での Docker サービス管理

```bash
# 全サービス確認
docker compose ps

# AI アプリのみ再起動
docker compose restart pptx-app

# Dify を停止してリソース確保
cd ~/dify && docker compose down

# Ollama の状態確認
curl http://localhost:11434/api/tags
```

## 7. 自社カスタム AI システムの全体像

### 最終的に目指すアーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│                    社内ネットワーク                        │
│                                                           │
│  [Oracle 12 DB サーバー]                                   │
│    ├── BOM / 受注 / 売上 / 在庫 / 出荷                     │
│    └── AI_AGENT ユーザー (SELECT 専用)                     │
│              ↕  python-oracledb                           │
│  [AI サーバー 192.168.1.10  WSL2 + Docker]                │
│    ├── FastAPI 共通 API 基盤                               │
│    │     ├── /api/analysis   (AI 分析)                     │
│    │     ├── /api/pptx       (PPTX 生成)                   │
│    │     ├── /api/excel      (Excel 生成)                  │
│    │     ├── /api/bom        (BOM 照会・分析)               │
│    │     └── /api/dashboard  (ダッシュボード用データ)        │
│    ├── Ollama (Qwen 3.5 / Llama)  ← GPU: RTX 5080        │
│    ├── React ダッシュボード（将来）                          │
│    └── Nginx リバースプロキシ（将来）                        │
│              ↕  HTTP / WebSocket                          │
│  [自席 PC 192.168.1.186]                                  │
│    └── Cursor + Remote-SSH → 開発・管理                    │
│              ↕                                            │
│  [社員の各 PC]                                             │
│    └── ブラウザ → http://192.168.1.10                      │
│         「先月の売上を分析して」→ AI が Oracle から取得       │
│         → 分析結果テキスト / PPTX / Excel をワンクリック出力  │
└─────────────────────────────────────────────────────────┘
```

### 開発ロードマップ

| Phase | 内容 | 時期目安 |
|-------|------|---------|
| 1 | PPTX Generator を会社サーバーにデプロイ | 直近 |
| 2 | Ollama + Qwen 3.5 の導入・テスト | Phase 1 と並行 |
| 3 | Oracle 連携 API の実装 | Phase 1 完了後 |
| 4 | BOM システムとの連携 | BOM 開発と並行 |
| 5 | React ダッシュボードの構築 | 中期目標 |
| 6 | Nginx + 認証基盤の整備 | 利用者拡大時 |

### 基本方針

> **「業務に特化したアプリを開発者が作って配る」**
>
> 汎用 AI プラットフォーム（Dify 等）を全員に使わせるより、ニーズに合ったカスタムアプリを FastAPI + AI で素早く開発し、ブラウザ経由で提供する。外注すれば数百万円の案件を、自社開発で柔軟かつ迅速に実現する。
